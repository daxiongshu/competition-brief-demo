1 · What the competition actually asks Mechanics

For every one of the 400 tasks drawn from the ARC‑AGI‑1 public training set, you submit a taskNNN.onnx network that must reproduce every train/test/arc‑gen input→output grid pair exactly (grids up to 30×30, one‑hot encoded across 10 colour channels as a [1, 10, 30, 30] tensor). A task scores max(1, 25 − ln(cost)) if — and only if — the network is functionally correct against the public benchmark and a hidden private benchmark (to block overfitting); an incorrect network scores 0. Cost, as of the current (April 30th) metric, is parameter count + memory footprint in bytes — MACs were removed entirely on May 4th after repeated profiler exploits (see the timeline below). Max total score across all 400 tasks is 10,000.

Submission is a submission.zip of up to 400 ONNX files, 100 submissions/day, banned ops include Loop, Scan, NonZero, Unique, Script, Function. Full details: competition page · official starter notebook · metric migration manual.

4 · Top kernels & the dominant technique pattern Kernels

Scores below are pulled live from Kaggle's scoring API for each notebook (fetch_kernel_score), not read off titles.

The winning pattern in the public kernel ecosystem is not a single strong model — it's a task‑level blend pipeline: one "base" 400‑task submission plus a rotating set of per‑task "override" ONNX files contributed by the community, re‑merged and re‑audited every few days. Kojimar's Task‑Level ONNX Baseline is the reference implementation of this merge step (it literally just overlays an overrides.zip onto a base_submission.zip, task by task). Downstream, notebooks like franksunp's Consolidated Audit run an A/B probe between competing community sources (Kojimar vs. Biohack44) and keep whichever wins per task — which is how it ties Kojimar at the top of this table.

A second, more technical lineage is ONNX graph surgery: seddiktrk's Graph Surgeon series loads a public submission, re‑scores every task locally with the same 25 − ln(memory + params) formula the host uses, and hand‑edits/fuses ONNX nodes per task to shave bytes while re‑verifying correctness against train/test/arc‑gen examples before accepting a change. boristown's Dark AGI Visualization Baseline pairs this with a full HTML "explainability" viewer over all 400 tasks (grids, ONNX op trace, and Python solver clues side‑by-side) — useful purely as a study aid even if you don't reuse its submission.

On the educational side, Chris Deotte (cdeotte) published both a layer‑by‑layer EDA series visualizing what trained ONNX graphs are actually doing on specific tasks, and a from‑scratch web GUI (built end‑to‑end by an agent) for hand‑authoring ONNX graphs — since open‑sourced by Clark Kitchen. Separately, MassimilianoGhiotto's Convolution Tutorial Series is the best plain‑language walkthrough of how Conv kernel size/padding/stride map onto ARC grid transformations.

7 · An actionable path to the top band Strategy

  1. Start from the host's starter — the official notebook plus the metric migration manual to get verify_network() / score_network() matching the live scorer exactly — don't trust an older local onnx‑tool pin (see the ONNX runtime‑compatibility thread for version pitfalls).
  2. Get to the ~6,200–6,400 band fast by studying/adapting a mid‑tier public blend such as konbu17's blend or octaviograu's hand‑built solvers — cheap, well‑understood territory.
  3. Climb to the ~7,150–7,185 plateau using the task‑level blend pattern: merge multiple public per‑task solvers and keep whichever wins per task, the way franksunp's audit does. Expect to be sharing this tier with ~800 other teams — it's necessary, not sufficient, for a medal.
  4. Break out of the plateau with graph surgery, not luck — study seddiktrk's graph‑surgery notebooks to learn which ONNX node patterns (Conv fusion, constant folding, redundant‑branch removal) shave real bytes without breaking correctness on train/test/arc‑gen. Re‑verify every change locally with the host's verify_network() before submitting — a single wrong task now costs you those points entirely (no partial credit).
  5. Spend your remaining 13 days on task‑by‑task iteration, not architecture searches — per Fritz Cremer's bucket analysis, cost is fairly evenly spread across all 400 tasks (bucket sums cluster 170–200 per 10‑task block) — there's no single "big task" to chase; gains come from consistently trimming many tasks a little. LLM‑agent‑assisted iteration (ChatGPT/Codex/Claude) is the community's dominant workflow — see the ChatGPT‑only approach and the more resource‑constrained free‑tier / multi‑LLM workflow for two different budgets.
  6. Consider merging — every team in the current top 10 has 2–5 members; solo teams are rare above 7,800 (see the live leaderboard). Check current submission‑count caps before merging late (the quota bug thread is worth a skim so you're not caught by a stale limit).
Honesty check: the plateau/ladder numbers above are all measured (leaderboard CSV + live per‑kernel score API), not estimated. Any number in a notebook title (e.g. "6425.76", "7113.80") was cross‑checked against the live scorer before being trusted in this report — a few titles were stale by 10–50 points relative to the kernel's current scored version.
